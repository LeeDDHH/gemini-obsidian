## **핵심 요약**

- **클로드 코드(Claude Code)가 “압도적”인 이유**는 단순히 모델 성능이 아니라, _에이전틱 개발을 전제로 한 실행 구조_ 때문이다.
    
- 사용자는 “가져올 페이지 하나 지정 → 그걸 참고해서 처리하세요 한 줄”만으로도, 미리 정의된 **파이프라인(MD 지침)** 이 동작해 **스크래핑 → 섹션 분해 → 컨벤션 정리 → 컴포넌트화 → 레지스트리 업데이트**까지 자동으로 완주하게 만든다.
    
- 특히 **서브 에이전트(Sub-agent)** 와 **스킬(Skill)** 기반의 “트리 구조 병렬 실행”이 가능해서, 커서 같은 툴이 _사용자가 직접 여러 번 호출해야 하는 작업_을 **한 번의 호출로 수십~수백 작업 단위로 분산 실행**하게 만든다.
    
- 대규모 작업(예: 773개 컴포넌트 등록, 125개 이미지 미구현분 일괄 생성, 최대 200개 단위 호출)이 가능한 이유는 **컨텍스트 엔지니어링**과 **스크립트화(코드 실행)로 컨텍스트 소모를 줄이는 설계**에 있다.
    

---

## **왜 “클로드 코드가 압도적일 수밖에 없는가” 구조적 이유**

  

### **1) “컨텍스트 엔지니어링”을 제품 기능이 받쳐준다**

- 초기 바이브 코딩은 “프롬프트 한 방”에 집중했지만, 지금은 **한정된 컨텍스트로 더 많은 일을 깊게 시키는 구조**가 핵심.
    
- 클로드 코드는 이를 위해 **서브 에이전트 / 스킬 / 파이프라인 문서** 같은 도구를 제공하고, 사용자는 “입력만” 주면 된다.
    

  

### **2) 서브 에이전트 = “에이전트가 에이전트를 호출”**

- 커서의 에이전트: **사용자가 에이전트를 호출**해야 함 → 대규모 분업을 하려면 반복 호출이 필요.
    
- 클로드 코드 서브 에이전트: **에이전트가 다른 에이전트를 호출**함 → 한 번의 명령으로 _트리 형태의 업무 분산_이 자동으로 일어남.
    
- 결과적으로 “조직 운영”처럼 동작: 한 명이 지시하면 여러 팀(서브 에이전트)이 병렬로 움직이는 형태.
    

  

### **3) “스크립트화”로 컨텍스트 소모를 극단적으로 줄인다**

- 대량 파일 생성/리소스 추출/스크린샷 처리 같은 작업을 LLM이 직접 하면:
    
    - 중간 산출물(이미지 payload, 텍스트 덩어리)이 **컨텍스트 윈도우에 남아** 비용/속도/지속성이 망가짐.
        
    
- 그래서 반복 작업은 **TypeScript 스크립트로 실행**시키고,
    
    - 에이전트는 “지시/판단/정리”에 집중
        
    - 스크립트는 “정해진 추출/생성”을 담당
        
        → **컨텍스트에는 결과만 남고 찌꺼기는 남지 않음**
        
    

  

### **4) 파이프라인을 문서로 정의 → “프롬프트는 트리거”**

- “MD 파일에 지침(어떤 에이전트가 어떤 순서로 무엇을 할지)”를 고정해두면,
    
- 사용자는 매번 긴 프롬프트를 쓸 필요 없이:
    
    - 이 URL 참고해서 처리해 같은 한 줄로 실행 가능
        
    
- 그래서 773개/125개 같은 “물량”이 가능한 자동화 루프가 만들어진다.

## **실제 워크플로우 패턴 (대화에서 드러난 운영 방식)**

1. **URL 지정**
    
2. **스크래핑/스크린샷/리소스 추출** (가상 브라우저 + 스크립트)
    
3. **섹션 단위로 분해**
    
4. **컨벤션 정리 + 컴포넌트화**
    
5. **레지스트리/페이지 컴포넌트 등 종합 결과물 업데이트**
    
6. **미구현 리스트 기반 배치 처리(예: 20개씩 반복, 총 125개 일괄 생성)**
    

---

## **“서브 에이전트 만드는 법” 요점**

- .claude/agents/ 같은 폴더에 **마크다운 파일 하나** 만들면 생성 가능
    
- 보통 구성은:
    
    - name / description / tools / model(옵션: opus/sonnet/haiku 등)
        
    
- 고급 사용자는 “에이전트 생성 자체”도 **스킬로 자동화**해서 계속 축적한다.
    
- 개선 방식은 반복 실행 후:
    
    - 자주 틀리는 규칙(폰트웨이트, 태그 추출 등)을 발견
        
    - 지침을 조금씩 추가 → **개인 최적화 에이전트로 진화**

## **인사이트**

- 클로드 코드의 강점은 “LLM이 똑똑하다”보다 **업무를 분해/병렬화/자동화하는 운영 체계**에 있다.
    
- 핵심은:
    
    - **문서화된 파이프라인**
        
    - **서브 에이전트 트리 호출**
        
    - **스크립트화로 컨텍스트 절약**
        
    - **반복 개선으로 개인화된 자동화 시스템 구축**
        
    
- 이 조합이 되면 “한 줄 프롬프트로 대규모 제작”이 현실이 된다.
