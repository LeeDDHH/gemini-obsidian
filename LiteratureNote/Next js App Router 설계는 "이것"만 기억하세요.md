# Next js App Router 설계는 "이것"만 기억하세요
https://www.youtube.com/watch?v=rX5b9rGiIww

## **요약**

- 핵심 원칙은 **Next.js App Router 철학**처럼, 가능한 한 **UI 렌더링과 데이터 패칭을 서버에서 처리**하는 것이다. 이렇게 하면 클라이언트로 내려가는 **JS 번들을 최소화**할 수 있어 성능 최적화에 유리하다.
    
- 따라서 **불필요한 JS를 클라이언트로 전송하는 것을 금지**하고, 서버에서 필요한 만큼만 처리하도록 원칙을 세운다.
    
- 또 “데이터 UI 변화 시점을 묶는다”는 말은, **데이터 변화와 UI 변화가 실제로 함께 움직이는 지점**을 기준으로 코드를 구성하자는 뜻이다. 즉 **도메인 단위로 응집도를 높이는 설계**다.
    
- 프론트엔드에서는 데이터가 바뀌면 UI가 따라 바뀌는 경우가 많으므로, 유연하게 대응하려면 **데이터 로직(패칭/가공)과 UI 컴포넌트를 분리하기보다 함께 묶는 편이 자연스럽다**는 결론에 도달한다.
    
- 이를 실현하는 방식이 **서버 컴포넌트**이며, 서버 컴포넌트 내부에서 **fetch와 render 로직을 결합해 도메인 단위로 응집**시키는 것이 “좋은 설계/좋은 코드”의 기준이 된다는 주장이다.
    

  

## **키워드 (최대 3개)**

- 서버 컴포넌트
    
- JS 번들 최소화
    
- 도메인 단위 응집도
