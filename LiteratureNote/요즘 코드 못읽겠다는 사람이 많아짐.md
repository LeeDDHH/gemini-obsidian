# 요즘 코드 못읽겠다는 사람이 많아짐
https://www.youtube.com/watch?v=9cXGI4E5Ll4

## **요약**

- 개발자는 **코드를 “짜는 시간”보다 “읽는 시간”이 더 길다.** (내가 쓴 코드, 남이 쓴 코드, 공식문서/라이브러리 코드 등)
    
- 그런데 요즘 **긴 코드를 보면 포기하거나**, 코드를 **영상처럼 “감상”** 하듯 대충 훑어보고 이해가 안 된다고 느끼는 사람이 늘었다.
    
- **문법만 알아서는 코드가 안 읽힌다.** 실무 코드는 대부분 **프레임워크/라이브러리 문법으로 구성**되어 있어서, 해당 생태계 사용법을 모르면 해석이 불가능하다.
    
    - 예: “자바는 아는데 서버 코드를 못 읽겠다” → 자바 실력 문제가 아니라 **스프링부트 지식 부족**
        
    
- **배경지식 부족도 독해를 막는다.**
    
    - 예: 파이썬+텐서플로우는 아는데 딥러닝 코드는 이해가 안 됨 → **딥러닝 개념(전제 지식)** 이 부족한 것
        
    
- 코드 읽기는 **왼쪽 위부터 한 단어씩 차근차근** 진행해야 한다.
    
    - 모르는 변수/함수/클래스가 나오면 **정의된 곳으로 점프해서 추적**
        
    
- 변수/함수/클래스 이름이 많으면 인간의 “뇌 RAM”이 부족해진다.
    
    - **중요한 이름은 메모/주석 추적**하면서 읽기
        
    - 작명이 쓰레기면 독해 난이도 급상승
        
    
- 가능하면 **코드 실행(로컬/테스트 서버)** 이 이해에 도움이 된다. 실행이 안 되면 더 어렵지만, 그래도 **내가 풀고 싶은 문제를 정확히 문장으로 적어두는 것**이 큰 도움이 된다.
    
- 이해가 안 되면 혼자 끙끙대지 말고 **동료에게 묻거나 작성자에게 질문**하는 게 빠르다.
    
- **집중력 문제**도 크다: 쇼츠 소비로 긴 글/긴 코드 집중이 어려워진다.
    
    - 화이트 노이즈, 강제 리딩(단어 단위로 빠르게 보여주는 확장 기능) 같은 방법을 제안
        
    
- “이 코드베이스 전부 갈아엎고 싶다”는 감정은 **아키텍처 문제가 아니라 ‘읽기 싫음/읽기 어려움’** 에서 오는 경우도 많다.
    
    - 읽고 분석해서 **리팩토링**하면 되는 문제일 수 있다.
        
    
- 결론: 코드를 쌓아올리는 것보다 **읽고 분석하는 능력이 더 중요**하니, **읽기 연습**을 하고 **쇼츠 같은 산만한 소비를 줄여라.**

## **실천 팁만 뽑아보기 (체크리스트)**

- “문법”보다 **핵심 프레임워크/라이브러리 사용법**부터 학습하기
    
- 코드 읽을 때 **왼쪽 위 → 한 토큰씩** + 모르는 심볼은 **정의로 이동**
    
- 중요한 변수/함수명은 **메모**해서 뇌 부담 줄이기
    
- 가능하면 **실행해서 입력/출력 흐름**부터 잡기
    
- 막히면 **질문**(동료/작성자)으로 빨리 병목 해소
    
- 집중력 장치(화이트노이즈/강제 리딩)로 **긴 텍스트 적응**
