## **요약 (Markdown)**

- **“RAG is dead”는 절반만 맞다.** 죽은 건 RAG 자체가 아니라, **문서를 잘게 쪼개서 임베딩만 믿는 ‘단순 RAG’**(naive chunking + vector search)다.
    
- RAG가 잘 안 되는 핵심 이유는 **문서를 자르는 순간 맥락이 날아가기 때문**이다. 예를 들어 “마케팅 비용 15% 증가”라는 조각만 있으면 **어느 회사/어느 분기/무슨 기준**인지 사라져 AI가 엉뚱한 답을 만들기 쉽다.
    
- 반대로 **Claude Code 같은 도구는 ‘전통적 RAG’보다 ‘AI가 직접 탐색(검색·열람·추가 탐색)’**하는 방식에 가깝다. 사람처럼 문서를 처음엔 일부만 읽고, 필요하면 더 읽으며 구조를 파악한다.
    
- 비유로는 **자율주행의 카메라 vs 라이다**:
    
    - RAG(라이다): 정밀해 보이지만 구축·운영 복잡도↑, 일부 문제로 전체가 흔들릴 수 있음
        
    - AI 직접 탐색(카메라): 단순하고 모델 발전과 함께 빠르게 좋아짐
        
    
- “제대로 된 RAG”를 하려면 단순 벡터 검색만으로는 부족하고, 최소한 다음이 필요하다:
    
    1. **하이브리드 서치**: 의미 기반(벡터) + 키워드 기반(BM25 등) 동시 사용
        
    2. **재랭킹/검수 단계**: 검색 결과를 다시 평가해 진짜 관련 조각만 추림
        
    3. (선택) **그래프 RAG**: 문서 간 인용/연관 관계를 미리 구축해 정확도를 올리지만 비용·유지보수 부담이 큼
        
    
- 최근 대안으로 **컨텍스트 윈도우 확대**가 있다. 문서가 많지 않거나 길지 않으면 **쪼개지 말고 통째로 넣는 게** 오히려 정확할 수 있다.
    
    - 단, **비용 증가**, **대규모 기업 데이터는 불가능**, **긴 문서 중간을 모델이 흘리는 현상(중간 성능 저하)** 같은 한계가 있다.
        
    
- 실용적 해법(현실적인 중간 지점):
    
    - 문서가 **적으면 통째로 컨텍스트에 넣기**
        
    - 문서가 **좀 있으면 ‘조각에 맥락 라벨’을 붙여 저장**(예: “A기업 2024 Q3 실적 보고서의 마케팅 지출 섹션 일부”)
        
    - 또는 **“요약본으로 검색 → 원문 단락으로 답변”**: 검색은 가볍게, 정답 생성은 원문으로 깊게 (도서관 ‘목록→본문’ 방식)

## **핵심 포인트 한 줄 정리**

  

**RAG가 죽은 게 아니라, “맥락 없는 chunk + 벡터검색”만으로 끝내는 방식이 한계에 도달했고, 이제는 하이브리드·재랭킹·맥락라벨·요약검색 같은 ‘정교한 RAG’ 또는 ‘AI 직접 탐색’으로 진화 중이다.**
