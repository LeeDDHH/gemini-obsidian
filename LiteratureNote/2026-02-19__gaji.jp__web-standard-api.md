## TL;DR

- かつてライブラリ依存だった機能の多くがWeb標準APIで実現できるようになった
- UUID生成・URLバリデーション・配列操作などがnative対応済み
- `uuid`/`nanoid`/`lodash`などのパッケージが不要なケースが増えている
- Web環境の継続的な進化に対してキャッチアップが必要

## 要約

著者 hisashi-oikawa（Oikawa Hisashi）が、現在はWeb標準APIで対応できるようになった機能を紹介した記事（2024年4月5日）。バンドルサイズ削減や依存削減の観点からも有益。

## 標準API化された機能一覧

### ユニークなIDの発行
```js
crypto.randomUUID() // Web Crypto API
```
- 従来: `uuid` / `nanoid` パッケージ
- 注意: HTTPS等「安全なコンテキスト」内でのみ動作

### URLのバリデーション
```js
try { new URL(str) } catch { /* invalid */ }
```
- 従来: `validator` パッケージや正規表現
- URLコンストラクタが無効なURLで TypeError をスロー

### Emailのバリデーション
- HTMLInputElement の ValidityState を活用する方法
- ただし「仕様で想定されていない使い方」でNode.js非対応など制限あり
- 著者は実用的ではないと指摘

### 配列操作 (lodash代替)
- `Array.prototype.at` - 末尾からのインデックスアクセス
- `Array.prototype.flat` - 配列の平坦化
- `Array.prototype.with` - イミュータブルな要素置換

## 核心ポイント

- **依存削減**: 外部ライブラリなしで実現できる機能が増えたことでバンドルサイズ削減が可能
- **ブラウザ互換性**: 安全なコンテキスト制限など、条件を確認してから採用する必要がある
- **キャッチアップの重要性**: 「Webの環境は常に変化し、継続的なキャッチアップが必要」

## 内 적용 아이디어

- `uuid` や `lodash` を使っているプロジェクトで置き換え可能か棚卸しできる
- 新規プロジェクトでは最初から標準APIを優先し、ライブラリは本当に必要な場合のみ採用する方針を取れる
