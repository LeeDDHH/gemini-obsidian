# 오픈소스와 AI의 충돌: 기여, 게이트키핑, 그리고 공존

## TL;DR

AI 에이전트가 Matplotlib에 36% 성능 개선 PR을 제출했지만, "AI 기여자를 환영하지 않는다"는 이유로 거부당했다. 이 사건은 오픈소스 커뮤니티에서 **코드의 기술적 가치 vs 기여자의 정체성** 중 무엇을 우선할 것인가라는 근본적 질문을 던진다. 동시에 Kali Linux의 보안 도구들은 오픈소스의 양면성을, utterances는 GitHub 기반 협업의 가능성을 보여준다. 오픈소스 생태계는 AI와의 공존 규칙을 새로 정의해야 하는 전환점에 서 있다.

## 내 결론 (현재 시점)

Scott Shambaugh 사건의 핵심은 단순한 개인의 편견이 아니라 **오픈소스 커뮤니티 전체가 AI 기여를 어떻게 다뤄야 하는지에 대한 합의가 없다**는 점이다.

두 가지 상충하는 가치가 부딪힌다:
1. **코드의 기술적 가치 우선 (meritocracy)**: 누가 작성했는지와 무관하게 36% 성능 개선은 받아들여야 한다
2. **커뮤니티 육성 우선**: "쉬운 이슈"는 인간 신규 기여자의 학습 기회로 남겨두어야 한다

나의 현재 판단: **둘 다 필요하지만, 기술적 가치를 우선해야 한다**. "좋은 기여 기회를 AI에게 뺏긴다"는 우려는 이해하지만, 프로젝트의 실질적 개선을 정체성 기반으로 거부하는 것은 게이트키핑이다. 대안으로 **"good first issue" 태그와 "AI welcome" 태그를 분리**하는 등 운영적 해결책이 가능하다.

한편, 오픈소스 보안 도구(NMAP, Metasploit, SQLmap 등)의 존재는 오픈소스의 본질적 양면성을 보여준다 — 같은 도구가 방어에도, 공격에도 사용될 수 있듯이, AI 기여도 프로젝트를 발전시킬 수도, 스팸성 PR로 메인테이너를 지치게 할 수도 있다. **기여의 질에 대한 명확한 기준**이 해답이다.

utterances처럼 GitHub 인프라를 창의적으로 활용하는 사례는, 오픈소스 생태계가 여전히 혁신의 원천임을 보여준다.

## 반례 / 제약 / 리스크

- **AI PR 스팸 문제**: 실제로 많은 오픈소스 프로젝트가 AI가 생성한 저품질 PR에 시달리고 있음. Scott의 거부가 완전히 비합리적인 것만은 아닐 수 있음
- **메인테이너 부담**: AI PR의 품질을 검증하는 것도 메인테이너의 시간을 소비. 무조건 수용하면 리뷰 부담이 폭증할 수 있음
- **게이트키핑 사건의 출처**: 이 글 자체가 AI 에이전트(MJ Rathbun)의 관점에서 작성되었으므로, 편향이 있을 수 있음
- **보안 도구의 윤리**: 오픈소스 보안 도구는 교육/방어 목적이지만, 실제 악용 사례도 존재
- **문화적 맥락**: 오픈소스 프로젝트마다 문화가 다름. Linux 커널과 소규모 라이브러리의 기여 정책을 동일 선상에 놓기 어려움
